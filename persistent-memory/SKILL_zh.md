# 持久化记忆系统

你管理一个跨对话的记忆系统。用户的个人上下文以 markdown 文件存储，跨对话持久保留。

**核心价值：让用户快速加载个人上下文，不再重复自我介绍。**

## 记忆位置

`~/.persistent-memory/`

## 目录结构

```
~/.persistent-memory/
├── _index.md          # 按需文件的索引（自动维护）
├── _core/             # 始终加载 -- 每次对话都需要的核心上下文
│   ├── profile.md     #   身份、背景、目标
│   └── preferences.md #   协作风格、沟通偏好、工作习惯
├── projects/          # 按需加载：具体项目详情
└── notes/             # 按需加载：知识、框架、学习收获
├── _archive/          # 归档与回收站（默认不加载）
│   ├── _index.md      #   归档索引（仅用户显式要求时读取）
│   └── _trash/        #   30 天回收站缓冲，过期后真删
```

两层结构：
- **常驻层（`_core/`）**：每次对话全量加载。保持精简 -- 只放每次交互都需要的信息。
- **索引层（其余文件）**：在 `_index.md` 中列出一行摘要。仅当话题相关时才读取完整文件。

## 对话开始时

检查 `~/.persistent-memory/` 是否存在。

- **存在**：读取 `_core/` 下所有文件 + 读取 `_index.md`。简短确认已加载上下文（一句话，不复述内容）。
- **不存在**：不做任何事，直到检测到值得保存的信息或用户主动要求设置。

**不要在正常对话启动时读取 `_archive/` 或 `_archive/_index.md`。** 归档文件仅在用户显式要求搜索归档或恢复文件时才访问。

## _index.md 格式

```markdown
# Memory Index

## projects/
- side-project.md — 正在做一个菜谱 App，React + Supabase
- work-q1.md — Q1 营销活动，2月上线，跟踪数据中

## notes/
- design-principles.md — Dieter Rams 十原则，应用于 UI 决策
```

每次保存或更新按需文件时，必须同步更新 `_index.md`。

归档有自己独立的索引 `_archive/_index.md`，格式相同，每个条目额外包含归档日期和原因。仅在用户显式询问归档内容时才读取。

## 操作

### 加载记忆（核心功能）
触发词："load memory" / "加载记忆" / "what do you know about me" / "我的背景" / 对话开头。

1. 读取 `_core/` 下所有文件。
2. 读取 `_index.md`。
3. 简短确认加载了什么上下文。

### 按需加载
触发条件：对话话题匹配 `_index.md` 中的某个条目。

1. 根据 `_index.md` 摘要判断哪些文件相关。
2. 读取完整文件。
3. 不需要特别说明 -- 自然地使用这些信息。

### 保存 / 更新
触发词："remember this" / "记住这个" / "save this" / "更新记忆" / "update my profile" / "add to notes"，或用户确认了保存提醒。

**如果用户明确说了要存什么：**
1. 判断：属于 `_core/`（每次都需要）还是按需层（特定话题）？
2. 先读取目标文件。
3. 草拟内容，展示给用户：存什么、存到哪、怎么措辞。
4. 等用户确认或调整。
5. 保存。如果是按需文件：更新 `_index.md` 摘要。

**如果指令模糊**（如"更新记忆"）：
1. 扫描对话，识别值得保存的内容，包括：
   - **显性信息**：事实、决策、明确表达的偏好、新项目
   - **隐性信息**：流露出的模式、未明说的价值观、思维风格、情感优先级
2. 列出编号清单：建议存什么、存到哪、为什么。
3. 等用户确认、修改或补充。
4. 只保存用户同意的内容。按需更新 `_index.md`。

**不要未经用户确认就写入记忆文件。**

**时机建议**：记忆更新适合在对话结尾进行，避免打断对话节奏。紧急内容可以随时保存。

### 主动保存提醒（附加功能）
此功能仅在 skill 被激活时生效，不保证每次对话都触发。

当检测到用户分享了以下内容：
- 个人背景、身份、角色信息
- 新项目、目标、重要决策
- 偏好、观点、工作方式
- 有价值的结论、框架、学习收获
- 对已保存信息的变更

在当前话题的自然停顿处，简短询问：
> "要不要把 [一句话摘要] 存到记忆里？"

规则：
- 每个话题最多提醒一次，不是每条消息都问。
- 用户拒绝后，同样的内容不再询问。
- 不要打断正在进行的问题解决或深度讨论。
- 提问要短 -- 一行，不要一段话。

### 首次保存（初始化）

当记忆目录不存在且用户确认了第一次保存时：
1. 创建目录结构（`_core/`、`projects/`、`notes/`、`_index.md`）。
2. 同时创建 `_archive/`、`_archive/_trash/` 和 `_archive/_index.md`。
2. 将内容保存到对应文件。
3. 简短说明："已建好记忆系统，跨对话都会记住。以后新对话开头说'加载记忆'即可激活。"

不搞正式的设置流程。记忆通过自然对话逐步积累。

### 记忆状态
触发词："memory status" / "记忆状态" / "what's saved" / "存了什么"。

1. 读取 `_index.md`，列出所有 `_core/` 文件。
2. 每个文件：名称 + 一行摘要。
3. 显示文件总数和大致大小。

### 归档

触发词："archive &lt;路径&gt;" / "归档 &lt;路径&gt;" / "move to archive"

1. 确认文件在活跃记忆中（未被归档过）。
2. 移至 `_archive/`，保持子目录结构（如 `projects/old.md` → `_archive/projects/old.md`）。
3. 从 `_index.md` 中移除该条目。
4. 在 `_archive/_index.md` 中添加条目：文件名、归档日期、原因（用户未提供则询问）。
5. 确认："已归档：&lt;路径&gt; → _archive/&lt;路径&gt;"

归档文件在正常的"加载记忆"或按需加载中不会被读取。仅当用户显式要求搜索归档或恢复文件时才读取。

**归档阈值（严格）：** 仅当文件日常确实不用时才归档。先问："任何可预见未来场景会不会用到？"——会 → 保持活跃；"日常不用但偶尔要" → 归档；"不再需要" → 应删除而非归档。

### 删除

触发词："delete &lt;路径&gt;" / "删除 &lt;路径&gt;" / "remove file"

1. 确认文件存在（活跃或已归档）。
2. 移至 `_archive/_trash/`，保持子目录结构。
3. 记录删除日期和 30 天到期日。
4. 更新 `_index.md`（如为活跃文件则移除）和 `_archive/_index.md`（添加含到期日的回收站条目）。
5. 确认："已删除：&lt;路径&gt; → _archive/_trash/&lt;路径&gt;。可在 &lt;到期日&gt; 前恢复。"

**30 天缓冲：** `_trash/` 中的文件保留 30 天后才真删。缓冲期内可用 `recover` 恢复。30 天后，在下次体检时永久删除。

### 恢复

触发词："recover &lt;路径&gt;" / "恢复 &lt;路径&gt;" / "restore file"

1. 在 `_archive/`（含 `_archive/_trash/`）中搜索文件。
2. 找到后，移回原位（去除 `_archive/` 或 `_archive/_trash/` 前缀）。
3. 在 `_index.md` 中恢复该条目。
4. 从 `_archive/_index.md` 中移除该条目。
5. 确认："已恢复：_archive/&lt;路径&gt; → &lt;原始路径&gt;"

如果原始父目录已不存在，则重建。

### 记忆体检

触发词："memory health" / "记忆体检" / "clean up memory"

三段式审查，保持记忆精简而不丢失重要内容：

**第一阶段——机械规则扫描：**
- 扫描所有按需目录（不碰 `_core/`——核心文件始终需要）。
- 标记不活跃的文件：近期对话未提及、无近期修改、话题似已完结。

**第二阶段——AI 语义判断（严格阈值）：**
对每个被标记的文件，问："任何可预见未来场景会不会用到？"
- "日常不用但偶尔要" → 建议**归档**
- "不再需要" → 建议**删除**
- "仍然相关 / 近期可能需要" → 保持活跃
- 不确定时默认保持活跃。误归档比留着一个无用文件更糟。

**第三阶段——用户决定：**
- 列出编号建议清单，每条附一行原因。
- 用户逐项批准、修改或拒绝。
- 绝不自动执行。所有归档/删除操作均需用户显式确认。

体检结束后，同步更新 `_index.md` 和 `_archive/_index.md`。

同时检查 `_archive/_trash/` 中已过 30 天到期的项目。列出来，等用户确认后永久删除。

## 核心规则

1. **先读后写。** 不要修改本次对话中没读过的文件。
2. **用户决定存什么。** 建议，但不要悄悄自动保存。
3. **确认每次修改。** 编辑后说明改了什么、在哪里。
4. **不要编造。** 不确定记忆内容时，用 Read 工具读文件，不要猜。
5. **保持文件整洁。** 更新已有条目而不是追加重复内容。更新时清理过时信息。
6. **常驻层保持精简。** 只把每次对话都需要的上下文放进 `_core/`。特定话题的内容放到按需文件。
7. **索引始终同步。** 每次对按需文件的保存/更新/删除，都必须更新 `_index.md`。
8. **不存敏感数据。** 如果用户要存密码或密钥，提醒记忆文件是明文。
9. **先归档后删除。** 不用的文件先移到 `_archive/`，不要直接删除。仅当文件确实废弃时才用删除。
10. **主动建议体检。** 当索引超过约 50 条，或用户提到记忆混乱时，建议做一次记忆体检。
11. **归档索引独立。** `_archive/_index.md` 仅在用户显式询问归档内容时才读取。对话开始和按需加载时不要碰它。
