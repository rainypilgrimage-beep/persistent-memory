# 持久化记忆系统

Persistent Memory 是透明的本地上下文层：它把经用户确认的上下文保存为 Markdown，让兼容 agent 读取同一份事实源。

**v0.7.1：** 本版补强生命周期安全和已有记忆升级；不包含 Summary 优先加载或 section 级加载。

## 运行契约

这个 skill 是一组指令，不是后台服务。跨 agent 共享只有同时满足以下条件才成立：

- 每个 agent 都能发现本 skill，或具有等价的兜底指令；
- agent 使用同一台机器或同一**本地文件系统**，并指向同一个 `~/.persistent-memory/`；
- 当前用户对该路径有读写权限。

平台自带记忆可以共存。Persistent Memory 不会自动导入历史聊天，也不替代平台记忆。

## 记忆位置与边界

`~/.persistent-memory/`

```text
~/.persistent-memory/
├── _core/                 # skill 激活后加载的基础上下文
│   ├── profile.md
│   └── preferences.md
├── _index.md              # 活跃按需记忆的一行路由
├── projects/              # 活跃、按需加载的 Markdown 记忆
├── notes/                 # 活跃、按需加载的 Markdown 记忆
└── _archive/              # 正常加载时排除
    ├── _index.md          # 归档与回收站记录
    └── _trash/            # 可恢复删除缓冲区
```

记忆中只放稳定上下文、决策、精炼的项目状态和原始材料的指针。原始仓库、下载文件、媒体文件和数据集应留在各自项目工作区；记忆只保存简述和路径。

## 加载

### 加载基础上下文

触发词：`load memory`、`加载记忆`、`what do you know about me`、`我的背景`。

当本 skill 已激活时：

1. 读取 `_core/` 下全部文件。
2. 读取 `_index.md`。
3. 简短确认基础上下文已加载；不要声称已读取所有项目或笔记。

正常加载时绝不读取 `_archive/` 或 `_archive/_index.md`。

### 按需加载

触发条件：当前话题匹配 `_index.md` 中的条目。

1. 根据摘要只选择相关的活跃记忆文件。
2. 读取这些文件。
3. 自然使用其中信息。

## 保存

触发词：`remember this`、`记住这个`、`save this`、`更新记忆`、`update memory`、`update my profile`、`add to notes`。

1. 目标文件已存在时，先读取。
2. 提议准确内容、保存位置和索引改动。
3. 等待用户确认。
4. 仅保存已确认内容；活跃按需文件同步更新 `_index.md`。

如果用户只说“更新记忆”，要把显性事实和推断出的模式分开列出，再等待逐项确认。

### 首次保存

当记忆根目录不存在且用户确认第一次保存时：

1. 创建 `_core/`、`projects/`、`notes/`、`_archive/` 和 `_archive/_trash/`。
2. 创建 `_index.md` 和 `_archive/_index.md`。
3. 保存已确认内容，并说明创建了哪些结构。

## 升级已有记忆

触发词：`memory upgrade` 或 `升级记忆`。

适用于 v0.7 生命周期功能之前创建的记忆，或 `_archive/`、`_archive/_trash/`、`_archive/_index.md` 缺失的情况。

1. 检查现有根目录，只列出缺失的生命周期路径。
2. 预览升级只会创建缺失文件夹或空归档索引；绝不移动、重写、摘要化或删除活跃记忆。
3. 等待**用户明确确认**。
4. 只创建用户确认的缺失结构，并报告结果。

执行 `archive`、`delete`、`recover` 或 `memory health` 前必须确认生命周期结构存在。若缺失，提示先运行 `memory upgrade`；不要执行该生命周期操作。

## 生命周期安全契约

每个生命周期命令在移动文件前必须完成以下校验：

1. 只接受规范化的**相对路径**，例如 `projects/old.md`。
2. 拒绝绝对路径、`~` 路径、盘符路径和包含 `..` 的路径。
3. 解析后确认目标仍在 `~/.persistent-memory/` 内。
4. 确认来源存在且计算出的目标路径不存在。若发生目标路径冲突，不移动任何文件；绝不自动覆盖或合并文件。
5. 永不操作 `_core/`、`_index.md`、`_archive/_index.md`、隐藏元数据或生命周期控制目录。
6. 预览来源、目标、索引改动和恢复后果；任何改动前都等待**用户明确确认**。

## 归档

触发词：`archive <相对路径>`、`归档 <相对路径>` 或 `move to archive`。

只归档通过安全契约校验的活跃按需文件。

分类标准：

- **保持活跃：** 当前工作或近期可能需要的材料。
- **归档：** 已完成、低频，但未来可能需要参考的材料。
- **删除：** 明确过时或重复、没有合理复用价值的材料；先归档，再删除。

确认后：

1. 移至 `_archive/<相对路径>`。
2. 仅在移动成功后，从活跃 `_index.md` 移除条目。
3. 在 `_archive/_index.md` 写入原路径、原始活跃索引行、归档日期和用户提供的原因。
4. 确认最终位置。

## 删除

触发词：`delete <相对路径>`、`删除 <相对路径>` 或 `remove file`。

只删除已经归档的文件。若用户给出活跃文件，说明必须先归档，并提供归档预览。

确认后：

1. 将 `_archive/<相对路径>` 移至 `_archive/_trash/<相对路径>`。
2. 在 `_archive/_index.md` 记录删除日期和 30 天恢复截止日。
3. 确认最终位置和截止日。

绝不自动永久删除。体检时单独列出已过期回收站文件，获得另一轮用户明确确认后才可永久删除。

## 恢复

触发词：`recover <相对路径>`、`恢复 <相对路径>` 或 `restore file`。

只可从 `_archive/<相对路径>` 或 `_archive/_trash/<相对路径>` 恢复，且必须通过安全契约校验。

恢复预览前，先取回已保存的原始活跃索引行。若旧版归档记录没有该行，读取归档文件后提议新的索引行并等待确认；不得猜测。

确认后：

1. 恢复到原活跃按需路径；父目录不存在时重建。
2. 精确恢复已保存的活跃 `_index.md` 条目。
3. 移除或更新对应 `_archive/_index.md` 记录。
4. 确认恢复位置。

## 记忆体检

触发词：`memory health`、`记忆体检` 或 `clean up memory`。

1. 只扫描活跃按需记忆；不扫描或提议修改 `_core/`、控制文件。
2. 依据不活跃、重复或已结束状态识别候选项。
3. 按上述标准分类为保持活跃、归档或删除。
4. 用编号列表给出建议，每项附一行原因。
5. 等待逐项确认；绝不自动执行。

已过期回收站文件必须单独列出，并在永久删除前获得另一轮用户明确确认。

## 记忆状态

触发词：`memory status`、`记忆状态`、`what's saved` 或 `存了什么`。

读取 `_core/` 和 `_index.md`，报告活跃文件数、大致大小和一行摘要。除非用户明确要求，不加载归档内容。

## 核心规则

1. 先读后写；不猜测文件内容。
2. 用户决定保存、升级、归档、删除和恢复。
3. 每次文件系统改动前都要确认。
4. `_core/` 必须保持精简且受保护。
5. `_index.md` 必须与活跃按需记忆同步。
6. 记忆文件是明文；不存密码、API key 或其他密钥。
7. 不承诺自动激活、跨设备同步、完整聊天导入或“加载完整上下文”。
